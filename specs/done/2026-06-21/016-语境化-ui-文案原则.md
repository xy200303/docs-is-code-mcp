# 语境化 UI 文案原则

## Meta

- status: done
- source: user-prompt

## 用户原始描述

补充 UI/UX 文案原则：文字应该结合语境和项目生成，不要 AI 化语言。

- UI/官网/组件文案必须从项目真实语境、用户任务、领域术语、README、代码、文档和实际功能中提取。
- 禁止默认生成 AI 式空泛语言、套话和万能营销句。
- 文案应说明具体对象、动作、状态、约束和真实下一步，而不是抽象形容词。
- CTA、标题、badge、空状态、错误提示、卡片描述都应可追溯到项目事实或用户确认。
- 增加文案自检：换一个项目名是否还能成立；是否使用抽象大词；是否缺少真实名词/动词；是否像模板站文案；是否有无法验证的承诺。

## 实际行为记录

- 记录来源：只能记录已阅读代码、已修改代码、测试结果或用户确认的事实。
- 功能全过程：按功能点记录触发入口、输入与前置状态、执行步骤、输出结果和副作用。
- 分支条件：完成后补充实际存在的正常、失败、边界、权限和状态分支，不只写一句结果。
- 默认参数行为：完成后补充源码里的默认值、配置来源、覆盖规则以及未传参数时的完整流程。
- 边界处理结果：完成后补充异常、空值、权限、状态等输入如何进入分支、在哪里返回、是否产生副作用。
- 验证结果：完成后记录验证命令、覆盖的流程分支和关联文件。
- 禁止事项：不要把猜测、常识或“看起来合理”的行为写成事实。

## 最终行为契约

1. UI/UX guidance 增加语境化文案原则
  - 条件：模型读取 ui-ux guidance 生成官网、组件、页面标题、CTA、badge、空状态、错误提示或卡片描述
  - 触发入口：spec_guidance_read 读取 name=ui-ux，或项目初始化/引导生成默认 specs/guidance/ui-ux.md 后由模型打开阅读
  - 输入与前置状态：projectRoot、specsDir、name=ui-ux；项目未覆盖时由 src/templates/guidance.ts 的默认模板生成
  - 执行过程：
    1. 在降低 AI 味章节补充文案必须结合项目语境生成
    2. 要求标题、CTA、badge、空状态、错误提示和卡片描述可追溯到项目事实或用户确认
    3. 要求文案说明具体对象、动作、状态、约束和真实下一步
    4. 加入文案自检：换项目名是否仍成立、是否使用抽象大词、是否缺少真实名词/动词、是否像模板站文案、是否有无法验证承诺
  - 输出结果：模型生成 UI 文案时会优先使用项目事实和具体任务语言，减少 AI 化泛文案。
  - 副作用：更新内置 guidance 模板和当前项目 specs/guidance/ui-ux.md；已有其他项目自定义 guidance 不会被覆盖
  - 默认行为：来源不明的文案应先确认或标记待确认，不使用泛化营销句补齐
  - 边界处理：如果文案换项目名仍成立、缺少真实名词/动词或包含无法验证承诺，应重写为项目具体对象、动作、状态和下一步
  - 结果摘要：ui-ux guidance 要求文案从项目语境、用户任务、领域术语、README、代码、文档、实际功能、状态和约束中提取，不写 AI 式套话或万能营销句。
  - 验证：bun test/unit.ts、bun test/smoke.ts、npm run verify
  - 关联文件：
    - `src/templates/guidance.ts`
    - `specs/guidance/ui-ux.md`
    - `test/unit.ts`
    - `test/smoke.ts`

2. active spec 模板加入文案可追溯检查
  - 条件：通过 spec_create 或 spec_bootstrap 生成 active spec
  - 触发入口：createSpecFromPrompt 调用 userPromptSpec，或 specTemplate 生成用户可编辑模板
  - 输入与前置状态：title、prompt、projectRoot、specsDir；prompt 可为 UI、官网、组件、交互或普通功能任务
  - 执行过程：
    1. 在 UI/交互质量检查加入语境化文案检查
    2. 要求标题、CTA、badge、空状态、错误提示和卡片描述来自用户任务、领域术语、README、代码、文档或实际功能
    3. 要求文案说明具体对象、动作、状态、约束和真实下一步
    4. 要求检查换项目名是否仍成立、是否使用抽象大词、是否缺少真实名词/动词、是否像模板站文案
  - 输出结果：生成的 active spec 会把文案语境和事实来源作为 UI 验收项。
  - 副作用：新生成的 active spec 文档更具体；已归档 done spec 不会被模板改动自动重写
  - 默认行为：非 UI 任务保留该章节作为条件性检查；UI 受影响时必须按项目语境填写
  - 边界处理：如果项目事实不足，模板要求先确认或搜索，而不是生成万能文案
  - 结果摘要：新 active spec 的 UI/交互质量检查会要求语境化文案、文案可追溯性和文案自检。
  - 验证：bun test/unit.ts、bun test/smoke.ts、npm run verify

  - 触发入口：spec_guidance_read 读取 name=quality-review，或模型在实现后按 Guidance Recommendations 自查
  - 输入与前置状态：projectRoot、specsDir、name=quality-review；当前实现包含 UI/交互/网页文案时适用
  - 执行过程：

    2. 检查文案是否说明具体对象、动作、状态、约束和真实下一步
